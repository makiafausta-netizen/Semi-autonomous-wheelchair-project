01 — Hardware Build

The Platform

Starting from a donated electric wheelchair from the Mary Campbell Center, I designed and wired a complete autonomous perception stack — from 12V lithium power distribution through edge compute to every sensor driver, written from the SDK up.

NVIDIA Orin Nano — edge inference & ROS2 host, runs all sensor nodes
Pico Flexx (PMD ToF) — 3D depth camera, streams point cloud at ~30 Hz via roypy SDK
RPLidar A1M8 ×2 — dual 2D spinning LiDAR for 360° environmental coverage
BNO08x IMU — 9-DOF, provides orientation quaternion + gyro yaw rate to ROS2
HC-SR04 ×4 — ultrasonic near-field ranging (FL, FR, BL, BR), each published as PointCloud2
ESP32 — MCU sensor hub; runs custom serial framing protocol bridging sensors to ROS2
MCP4728 DAC + CD4066 analog switch — motor control interface from compute to drive electronics
Power architecture

Two 12V 54Ah Dakota Lithium batteries feed a circuit breaker, then a DC-DC converter steps down to 5V for the compute stack. All wiring, electrical housing, and cable routing designed and built by hand.

Every ROS2 node in this stack was written from scratch. The pipeline converts raw hardware signals into typed ROS2 messages, runs real-time fusion, and outputs velocity commands to the motor controller. I wrote four different versions of the LiDAR driver alone as I iterated through issues with angle conventions, coordinate frames, and dual-sensor splitting.

I also shipped both a ROS1 and ROS2 version of the PMD depth camera node, migrating from the rospy-based DualPicoSubscriberNode to the ROS2 FlexxCameraNode maintaining identical topic names across versions.

ROS2 · Python

dual_rplidar_node.py

Connects to two RPLidar A1M8s over serial (/dev/lidar_left, /dev/lidar_right), one thread per sensor. Negates raw angles to flip from CW to CCW, applies per-sensor FOV gating (0°–180° / 180°–360°), converts polar → Cartesian, and publishes PointCloud2. Includes auto-reconnect on serial drop.

/rplidarpc_left /rplidarpc_right angle negation FOV gate auto-reconnect
ROS2 · roypy SDK

flexx2_camera_node.py

Direct hardware driver for the pmd Flexx2 via the Royale SDK. Registers a FlexxListener callback, queues frames asynchronously, transforms from Royale frame (X=right, Y=down, Z=forward) to ROS base_link (X=forward, Y=left, Z=up), then filters a 0.2–1.0m × ±0.3m detection box and publishes cleared/blocked status based on point density threshold (>50 points).

ROS2 · Serial UART

esp32_bridge_node.py

Serial bridge running a background thread that reads framed messages from the ESP32. Routes on prefix tag: IMU_G buffers gyro-Z, IMU_Q merges quaternion + buffered gyro into a single sensor_msgs/Imu, and US publishes four ultrasonic readings as Int32MultiArray.

/imu_data /ultrasonic gyro-merge trick prefix routing
ROS2 · PointCloud2

ultrasonic_pointcloud_node.py

Converts the raw Int32MultiArray (cm) from the ESP32 bridge into per-sensor PointCloud2 messages. Each reading becomes a single 3D point at [distance_m, 0, 0] in that sensor's URDF frame — the TF tree in robot_state_publisher handles rotation into the world frame automatically. Invalid (≤0) or sub-min-range readings are dropped, not published.

03 — Interesting Component

The ESP32 Hardware Bridge
& Custom Serial Protocol

Operating Principle

ESP32 → ROS2 · UART 115200 · Prefix-routed framing · Gyro-merge trick

The most technically interesting component I built is the ESP32Bridge. A custom hardware abstraction layer that translates raw microcontroller sensor data into properly typed ROS2 messages over a single UART link. Off-the-shelf serial parsers couldn't handle the heterogeneous mix of IMU quaternions, gyro rates, and four ultrasonic distances I needed to multiplex simultaneously. 

# Wire format — self-framing, prefix-routed
<IMU_Q, 0.12, 0.45, 0.78, 0.32>  ← orientation quaternion (x,y,z,w)
<IMU_G, 0.042>                    ← yaw rate from gyro (rad/s)
<US, 45, 23, 67, 12>             ← 4 ultrasonic distances in cm

The most elegant problem I solved here is the gyro-merge trick. ROS2's sensor_msgs/Imu requires orientation and angular velocity in a single message — but the ESP32 sends them as separate frames to minimize packet size and keep firmware simple. My bridge node buffers the latest gyro_z value and stamps it onto the next quaternion frame before publishing. The result is a clean, complete IMU message with zero data loss and no extra bandwidth. 

elif prefix == "IMU_G":
    # Buffer gyro-Z — merged on next quaternion frame
    self._gyro_z = float(parts[1])

elif prefix == "IMU_Q":
    imu_msg.orientation.x = float(parts[1])
    imu_msg.orientation.y = float(parts[2])
    imu_msg.orientation.z = float(parts[3])
    imu_msg.orientation.w = float(parts[4])
    # Attach buffered gyro — one complete Imu message
    imu_msg.angular_velocity.z = self._gyro_z
    self.imu_pub.publish(imu_msg)

elif prefix == "US":
    msg.data = [int(parts[1]), int(parts[2]),
               int(parts[3]), int(parts[4])]
    self.ultrasonic_pub.publish(msg)

The covariance matrices in the published Imu message are carefully set: X/Y angular velocity and all linear acceleration terms are marked with 99999.0 (effectively infinite uncertainty), which signals any downstream EKF to ignore those axes. Only yaw rate and orientation quaternion are trusted. This prevents a noisy or absent accelerometer from polluting localization.

The ultrasonic-to-PointCloud2 conversion in the companion node is another key design choice: rather than writing a custom message type, I map each distance reading to a single 3D point [dist_m, 0, 0] published in that sensor's URDF frame. The TF tree in robot_state_publisher rotates it into world coordinates automatically — the navigation stack sees it as just another point cloud, no special casing required.

This component is directly analogous to WindBorne's sensor manufacturing pipeline: custom communication protocols, careful handling of timing artifacts, covariance modeling for sensor trust, and designing data pipelines that are simple enough to be reliable in the field.

04 — Design for Ease of Use

3-Zone Obstacle Avoidance
with Hysteresis

Why hysteresis is the critical design choice

Without hysteresis, obstacle avoidance systems oscillate — the wheelchair spins back and forth at a threshold boundary, which is disorienting and alarming to a user with a disability. I solved this with two deliberately asymmetric thresholds: avoidance activates at 20 cm but only clears at 35 cm. This 15 cm dead-band means the system commits to an evasion action before reversing it, producing smooth, predictable behavior.

The rotation direction is also chosen algorithmically: the node computes left_clear and right_clear by taking the minimum distance from the LiDAR and depth camera on each side, then rotates toward whichever half-space has more room. Users consistently get turned toward open space, not toward a wall.

3-zone control logic
ZONE 1 — Clear
Full speed · proportional heading control
> 1.2 m
ZONE 2 — Caution
Reduced speed · turn toward clearer side
0.55 – 1.2 m
ZONE 3 — Stop
Full stop · rotate in place
< 0.55 m
Hysteresis thresholds
ACTIVATE: 20 cm · CLEAR: 35 cm · dead-band: 15 cm

Sensor fusion — minimum-distance across all sources

The SensorAvoidance node subscribes to all five data streams: /rplidarpc_left, /rplidarpc_right, /flexx2/filtered_points, and all four ultrasonic point clouds. For each zone it takes the minimum distance across all active sensors. Any one sensor detecting danger triggers the response — a sensor dropout just means slightly reduced coverage, not a blind spot.

RViz live visualization — all streams simultaneously

All sensor streams publish standard PointCloud2 messages, which means they appear in RViz immediately with no custom plugin. White cloud = Pico Flexx depth camera. Yellow markers = ultrasonic distance readings as 3D points in each sensor's URDF frame. The full wheelchair URDF and robot_state_publisher render the model in-situ. Field debugging becomes visual, not log-parsing.

MATLAB simulation — validated before hardware

Before deploying to the real wheelchair, I tested my sofwtware, hardware and logic through MATLAB using a 3D STL model of the actual wheelchair, 42 ray-cast fake LiDAR beams (21 per pod) using slab-method ray-box intersection, and color-coded rays (green/yellow/red). The simulation ran the identical state machine and thresholds as the real SensorAvoidance node.

05 — Simulation

MATLAB Validation Before
Any Hardware Deployment

The discipline of validating algorithms in a simulation environment before trusting them in the field is central to this project. I built a complete MATLAB simulation to animate the wheelchair STL model, translating and rotating correctly — with two simulated LiDAR pods casting 42 rays total using slab-method ray-box intersection.

Rays are color-coded in real time: green when distance exceeds the caution zone, yellow in the caution range, red when within the danger threshold. The 3-zone state machine governing the simulation is the exact same logic as the deployed ROS2 node — not a simplified proxy. This means the thresholds, zone boundaries, and rotation direction logic were all proven in simulation before touching the real motor controller.

This simulation-first mentality is directly analogous to flight-testing instrumentation: you don't deploy an untested sensor calibration to a balloon — you validate it in the lab first.

06 — Engineering Problem

The ROS1 → ROS2 Migration

The PMD depth camera driver was originally written in ROS1 (rospy). As the project matured and the NVIDIA Orin Nano's full capabilities required ROS2, I had to migrate the entire stack across framework versions without breaking the downstream navigation nodes.

The constraint I set for myself: keep the same topic names, node names, and message types so that SensorAvoidance. I iterated through four versions, as I was learning which version was best for real time data analysis and perception, before landing on the final pure-ROS2 architecture.

v1

Empty ROS2 package shell

Created the ROS2 package structure and topic scaffolding. Navigation stack pointed at new topics while the old ROS1 nodes still ran.

v2

ROS1 Bridge relay

Installed ros1_bridge to relay messages between frameworks. Allowed running both simultaneously for A/B comparison.

v3

Native ROS2 node porting

Rewrote each ROS1 node in rclpy rospy.Publisher. Maintained identical topic names and message types.

v4

Full ROS2, bridge removed

Eventually if the bridge is dropped the bridge entirely. ROS1 and ROS2 can still communicate if needed, interfaces are stable.

Built a full MATLAB simulation with identical 3-zone control logic before touching hardware. Equally tested hardware sensor data in the simulation environment.
